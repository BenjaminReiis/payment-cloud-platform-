03 · objetivo

O que a plataforma faz

Uma base realista para uma plataforma de pagamentos moderna — cobrindo o ciclo completo de usuário, cliente e cobrança.

✓

Cadastro e autenticação de usuárioRegistro com Bcrypt para hash de senha e login com emissão de JWT.

✓

Proteção de rotasMiddleware de autenticação bloqueia acesso a rotas sensíveis sem token válido.

✓

CRUD parcial de clientesCadastro e gerenciamento de clientes vinculados ao usuário autenticado.

✓

CRUD parcial de pagamentosCriação de cobranças e atualização de status (pendente, pago, falhou).

✓

Dashboard com resumo financeiroVisualização agregada das métricas de pagamentos do usuário.

✓

Arquitetura separadaFrontend, backend e banco de dados isolados, cada um em seu próprio container.

05 · uso

Subindo o ambiente completo

Um único comando levanta backend, frontend e banco de dados em containers isolados.

terminal
# clone o projeto e entre na pasta
cd cloudpay-platform

# copie as variáveis de ambiente de exemplo
cp backend/.env.example backend/.env
cp frontend/.env.example frontend/.env

# suba todos os serviços (api + frontend + postgres)
docker compose up --build

# aplique as migrations do Prisma (em outro terminal)
docker compose exec backend npx prisma migrate dev
fluxo da requisição — exemplo
// auth.routes.ts
router.post("/login", async (req, res) => {
  const { email, password } = req.body;
  const result = await authService.login(email, password);
  res.json(result); // { token, user }
});

// middlewares/auth.ts
export function authMiddleware(req, res, next) {
  const token = req.headers.authorization?.split(" ")[1];
  if (!token) return res.status(401).json({ error: "unauthorized" });
  const payload = verifyJwt(token);
  req.user = payload;
  next();
}
i

Após subir os serviços, a API fica disponível em http://localhost:3000 e o frontend em http://localhost:5173 (ajustável via variáveis de ambiente).
